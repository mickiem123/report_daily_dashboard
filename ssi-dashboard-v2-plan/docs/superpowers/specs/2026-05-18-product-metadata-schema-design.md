# Product Metadata Schema Design

Date: 2026-05-18
Status: Approved for implementation planning

## Goal

Redesign the dashboard structure so products, headline metrics, normal metrics, and sub-product metrics are configurable from the frontend while keeping the current `daily_metrics`, `weekly_metrics`, and `monthly_metrics` value tables.

## Current Problem

The app currently renders six product cards from fixed TypeScript extractors and fixed Supabase columns. Adding a new product card or metric requires a developer to update the database schema, TypeScript row type, field labels, validation lists, extractors, and tests.

The new design keeps the current wide metric tables for value storage but moves product/card structure into metadata tables. Users can add product cards and metric columns in the dashboard UI.

## Decisions

- Keep `daily_metrics`, `weekly_metrics`, and `monthly_metrics`.
- All three metric tables must stay schema-identical.
- New metric columns are numeric only in v1.
- A Product is one dashboard card.
- Each Product has exactly one headline metric.
- A Product can have normal metrics.
- A Product can have sub-products, and sub-products contain normal metrics only.
- Frontend users can add product cards and metric columns.
- Frontend users can delete metric columns after confirmation.
- Deleting a metric drops its physical column from all three metric tables and deletes or archives its metadata.
- Deleting a product is allowed only when the product has no metrics.
- Hide/show, rename, reorder, and moving existing metrics are deferred from v1.
- Browser code must not receive a Supabase `service_role` key or any database-owner credential.

## Data Model

### Existing Value Tables

The existing tables remain the source of metric values:

- `public.daily_metrics`
- `public.weekly_metrics`
- `public.monthly_metrics`

Each table keeps `ngay` as the date key. New user-created metric columns are added to all three tables as nullable `numeric` columns.

### `products`

One row per card.

Fields:

- `id uuid primary key`
- `key text not null unique`
- `name text not null`
- `headline_metric_id uuid null`
- `sort_order integer not null`
- `is_visible boolean not null default true`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Rules:

- `key` is generated from `name`.
- `headline_metric_id` points to one metric in `metric_definitions`.
- If a product has no headline metric, the card renders an empty headline state.

### `sub_products`

Optional groups inside a product.

Fields:

- `id uuid primary key`
- `product_id uuid not null references products(id)`
- `name text not null`
- `sort_order integer not null`
- `is_visible boolean not null default true`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Rules:

- A sub-product belongs to exactly one product.
- Sub-products are used only for grouped normal metrics.

### `metric_definitions`

One row per metric column.

Fields:

- `id uuid primary key`
- `column_key text not null unique`
- `label text not null`
- `unit text not null`
- `product_id uuid not null references products(id)`
- `sub_product_id uuid null references sub_products(id)`
- `placement text not null`
- `sort_order integer not null`
- `is_visible boolean not null default true`
- `is_percent boolean not null default false`
- `is_inverse boolean not null default false`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Rules:

- `placement` is one of `headline`, `normal`, or `sub_product`.
- `column_key` is generated by the backend from `label`.
- `unit = '%'` sets `is_percent = true`.
- `placement = sub_product` requires `sub_product_id`.
- `placement = headline` clears or demotes the previous product headline.
- Existing hardcoded columns are backfilled into this table during migration.

## Backend API

Schema-changing actions go through a trusted backend boundary. The recommended implementation is one Supabase Edge Function named `schema-admin`.

### `add_product`

Input:

- `name`

Behavior:

- Generate a safe unique product `key`.
- Insert a row into `products`.
- Return the created product.

No metric table columns are created.

### `add_metric`

Input:

- `label`
- `unit`
- `product_id`
- `placement`
- optional `sub_product_id`
- optional `sub_product_name`
- optional `is_inverse`

Behavior:

- Generate a safe unique `column_key` from `label`.
- Validate the generated column does not already exist in any metric table.
- If `placement = sub_product` and `sub_product_name` is provided, create the sub-product first.
- Run `ALTER TABLE ... ADD COLUMN <column_key> numeric` on:
  - `public.daily_metrics`
  - `public.weekly_metrics`
  - `public.monthly_metrics`
- Insert `metric_definitions`.
- If `placement = headline`, set `products.headline_metric_id` to the new metric.
- If the product already had a headline metric, change the old metric placement to `normal`.
- Return the created metric definition and affected product.

### `delete_metric`

Input:

- `metric_id`
- confirmation value derived from the metric label or column key

Behavior:

- Load metric metadata.
- Verify the confirmation value.
- Run `ALTER TABLE ... DROP COLUMN <column_key>` on all three metric tables.
- Delete or archive the metric metadata.
- If it was the product headline, clear `products.headline_metric_id`.
- Return success.

### `delete_product`

Input:

- `product_id`
- confirmation value derived from product name

Behavior:

- Reject if any metric definitions still belong to the product.
- Delete the product metadata.
- Return success.

## Frontend UI

The structure editing UI is contextual and blended into the existing dashboard.

### Add Product Card

At the end of the card grid, render a subtle `+ Add Card` tile.

Flow:

1. User clicks the tile.
2. Dialog asks for product name.
3. Frontend calls `schema-admin` with `add_product`.
4. Product config refetches.
5. New empty card appears in the grid.

### Add Metric

Each product card has a small contextual `+` control.

Flow:

1. User clicks `+` inside a card.
2. Dialog opens scoped to that product.
3. User enters metric display/header name.
4. User selects placement: headline, normal, or sub-product.
5. User selects unit: `tỷ`, `KH`, `HĐ`, `tài khoản`, or `%`.
6. If placement is sub-product, user chooses an existing sub-product or creates one inline.
7. Frontend calls `schema-admin` with `add_metric`.
8. Data and metadata refetch.
9. The new metric appears in the card and data table with empty values for existing dates.

### Delete Metric

Each metric row on the card back has a delete control.

Flow:

1. User clicks delete on a metric row.
2. Dialog explains that the metric column and historical values will be deleted from daily, weekly, and monthly tables.
3. User confirms.
4. Frontend calls `schema-admin` with `delete_metric`.
5. Data and metadata refetch.
6. The metric disappears from the card and table.

### Delete Product

Each product card has a delete control.

Flow:

1. User clicks delete on a product card.
2. If the product still has metrics, the app blocks deletion and tells the user to delete metrics first.
3. If the product has no metrics, user confirms.
4. Frontend calls `schema-admin` with `delete_product`.
5. Product config refetches.
6. The card disappears.

## Rendering Changes

The app stops treating product structure as hardcoded extractor output.

Required rendering changes:

- Fetch metadata alongside metric rows.
- Build `ProductCard[]` from metric rows plus metadata.
- Replace static `NUMERIC_FIELDS` with visible `metric_definitions`.
- Replace static `FIELD_LABELS` with metric labels from metadata.
- Keep pure compute helpers for formatting, diff, trend, and strong-move detection.
- Keep the current card front/back visual pattern.

## Security

- The frontend never receives privileged database credentials.
- The Edge Function validates all input before running DDL.
- Generated column names must be normalized and allowlisted.
- SQL identifiers must be quoted safely on the backend.
- Metadata tables allow public `SELECT` for dashboard rendering and deny direct public `INSERT`, `UPDATE`, and `DELETE`; metadata writes happen through `schema-admin`.
- DDL actions are available to anyone who passes the current dashboard gate in v1.

## Validation Rules

`add_metric` rejects:

- empty label
- unsupported unit
- unsupported placement
- sub-product placement without a sub-product
- duplicate generated column key
- generated column key that conflicts with reserved fields like `id`, `ngay`, or `pushed_at`

`delete_metric` rejects:

- missing metric
- confirmation mismatch
- attempts to delete reserved fields

`delete_product` rejects:

- missing product
- confirmation mismatch
- product with any attached metrics

## Testing Strategy

Unit tests:

- column key generation from Vietnamese and mixed text labels
- config-to-card builder for headline, normal, and sub-product metrics
- headline replacement behavior
- add/delete request validation

Data layer tests:

- metadata fetch maps products and metrics correctly
- dynamic grid columns render from metadata
- add metric calls backend with expected payload
- delete metric requires confirmation

Backend tests or smoke checks:

- `add_metric` adds the same numeric column to all three metric tables
- `delete_metric` drops the same column from all three metric tables
- duplicate column creation is rejected
- product deletion is rejected when metrics exist

Manual verification:

- Add a product card from the dashboard.
- Add a headline metric to that card.
- Add a normal metric to that card.
- Add a sub-product metric to that card.
- Enter values in the data grid and save.
- Delete a metric after confirmation.
- Confirm all three metric tables still have matching schemas.

## Deferred From V1

- Hide/show product and metric controls.
- Rename product or metric.
- Reorder products or metrics.
- Move existing metrics between products or sub-products.
- Non-numeric metric columns.
- Fully normalized metric value storage.
- Cascade-delete product with all metric columns.
